# Авто рассылка СМС - Архитектурные решения (ADR)

## ADR-0001: Выбор `uv` как пакетного менеджера

**Дата:** 2026-04-29
**Статус:** Принято

### Контекст
Нужен менеджер зависимостей и виртуальной среды для Python 3.11+ проекта, который собирается в `.exe` через PyInstaller. Требуется детерминированная установка (lock-файл) и быстрое разворачивание dev-окружения.

### Альтернативы
- `poetry` — зрелый, но медленнее и тяжелее
- `pip + venv` — стандарт, но без lock-файла из коробки
- `pdm` — гибкий, но менее распространён

### Решение
Использовать `uv` — единый инструмент для виртуальных сред, lock-файлов и установки. Скорость на порядок выше альтернатив, lock-файл `uv.lock` фиксирует все транзитивные зависимости.

### Последствия
- `uv.lock` хранится в репозитории
- Команды в документации и `README.md` используют форму `uv run …`
- Конечному пользователю Python не нужен (всё пакуется в `.exe`), но dev-окружение требует установленного `uv`

## ADR-0002: Выбор `customtkinter` для GUI

**Дата:** 2026-04-29
**Статус:** Принято

### Контекст
Нужен desktop-GUI для Windows с современным внешним видом, минимумом внешних рантаймов и простой упаковкой в один `.exe`-файл.

### Альтернативы
- `PyQt6` / `PySide6` — мощно, но большой бандл и более сложная лицензионная модель
- Стандартный `tkinter` — устаревший вид, требует ручной кастомизации
- `Dear PyGui` — игровой стиль, не подходит под бизнес-инструмент
- Web-обёртка через `pywebview` — лишняя зависимость от браузерного движка

### Решение
Использовать `customtkinter>=5.2.0` — современный вид поверх стандартного Tk, ноль внешних рантаймов, маленький бандл PyInstaller.

### Последствия
- Темизация задаётся через `set_appearance_mode("dark")` и `set_default_color_theme("blue")` в `app.py`
- Кастомизация виджетов ограничена в сравнении с Qt — это приемлемо для wizard из 4 простых экранов
- Дополнительная зависимость в `pyproject.toml`: `customtkinter>=5.2.0`

## ADR-0003: Прямой вызов `adb.exe` через subprocess

**Дата:** 2026-04-29
**Статус:** Принято

### Контекст
Для слоя `src/adb/` нужно общаться с подключённым телефоном (поиск устройств, отправка SMS, чтение статусов). В Vision были оставлены оба варианта: библиотека `pure-python-adb` или системный `adb.exe` напрямую через subprocess.

### Альтернативы
- `pure-python-adb` — Python-клиент к adb-серверу. Плюсы: типизированный API, работа без shell. Минусы: дополнительная зависимость, всё равно нужен запущенный `adb-server`, неактивный мейнтейнинг (последний релиз 2020 г.)
- `subprocess` + `adb.exe` из `bin/adb/` — нативный путь. Плюсы: ноль зависимостей, актуальная версия из platform-tools, любые adb-команды работают без обёртки

### Решение
Использовать `subprocess.run` поверх `adb.exe`, упакованного в `bin/adb/`. Поиск исполняемого файла — через `find_adb_executable()`: сначала `sys._MEIPASS` (PyInstaller-бандл), потом dev-путь.

### Последствия
- В `pyproject.toml` не добавляется `pure-python-adb` (минус одна зависимость, минус CVE-поверхность)
- Все `adb`-вызовы запускаются с `creationflags=CREATE_NO_WINDOW` (Windows-only), чтобы не мелькало окно cmd
- Каждый вызов ловит `TimeoutExpired`/`OSError` → `AdbError`, отсутствие `adb.exe` → `AdbNotFoundError`. Высокоуровневые функции (`get_connection_status`) превращают исключения в `ConnectionStatus` для GUI
- Кросс-платформенность не закладывается — `CREATE_NO_WINDOW` существует только на Windows (соответствует Vision §«Среда: локальный Windows ПК»)
